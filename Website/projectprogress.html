Description

Description When we first started this project, we began with what our project was going to be. As a group we chose a mobile fitness application because we felt that during the current circumstances with the COVID-19 pandemic that many peoples health and fitness would be poor. This application would be beneficial for us as a group as well as the general public. After we had decided on the topic of this project, we transitioned to the planning phase, deciding on the goals of the project as well as the scope which would be met with specific features to achieve the goals.

Our planning phase was not too long, we managed, as a group, to decide on which features to implement into our application relatively quickly. The biggest challenge however was that all of us had not done a mobile application before and we lacked the knowledge and experience required to actually implement the features and to estimate the time required to do so. After some discussion, we had formulated a list of features that would exist in a working prototype of this project, which included some extra features if time permitted to implement them.

Initially the group did not progress compared with the plan which had been created as the group was confused with what was required for the assignments and didn’t realise that coding was meant to be done across the whole duration of the course and not just for the third assignment. However once that confusion had been rectified, the group, for the most part, was able to get their work done according to a more condensed plan.

Dead-ends
In regards to social sharing, when the share button is pressed it would be ideal to send it to the specific social media platform (e.g. Facebook), but this requires an API, which we couldn’t use as it costs money.

Pivotal decisions made
A pivotal decision made was the platform on which we wanted our application to be on, in the end we decided on Android. This is mainly because it is much easier to code an application with, as well as the majority of the group having android phones, making it easier to do testing. Another important decision made was the software used for development of the application; as a group we chose Android Studio. We chose Android Studio because it is well known and the language in which to code in is Java, which the group all had experience doing.

Any changes to the original project plan.
Some deviations were made to the original plan in regards to the calorie counter, due to time constraints it was originally going to be coded with one example, however upon further talks with the tutor, they requested that more examples be provided.

Areas of work that the team would have liked to have done but couldn’t due to time constraints.

We would have liked to implement some of the extended features, such as the rewards system, that the group had discussed, however due to time constraints this was not possible. A dark or alternate colour scheme for the application would have also been implemented for people with visual impairment.

For the Recipe MVF, the functionality would have preferred to be different than it currently is. Instead of tapping on the recipe to display it, swiping on that recipe would have been the ideal method for viewing it.

Scope creep

We had strong project management and team communication since the start of the project which helped us extensively with our project. We did not have to make any major changes with our scope of the project. However, the minor changes we had with the app were related to the API’s as we were not able to purchase them. Due to this we could not implement the social media API keys and the Location API key for the Social Sharing MVF and the Exercise and Geolocation EVF, respectively.

Windfall

Most of us were able to finish our assigned MVF design and coding quicker than expected through Android Studio. We originally estimated that we would finish the coding of each MVF after about 20-30 hours of work but we were pleasantly surprised when we managed to complete them by around the 10-20 hour mark, which allowed us to double check each other's MVFs with the extra time left over. The main component of coding that most of us were able to complete faster than expected was the design aspect. Coding the interface and layout of each page was a lot easier than expected and gave us more time to perfect the interactions and functionality of each page.

Tools&Technologies

Android Studio
https://developer.android.com/studio
Android Studio is an official development environment used for designing and developing applications for the Android operating system. It is quite a useful tool as it can be used on Windows, MacOS and Linux systems, meaning that each group member has easy access to the tool. We'll be using the latest version of Android Studio (4.0.1) and is free to download and use on all platforms.

Visual Studio Code
https://code.visualstudio.com
We will likely be using Visual Studio Code again for the development of our website, VSC will be used for the HTML and CSS coding aspect of our website. It is easy to use and automatically formats code when needed, we'll be using the latest version of Visual Studio Code for both Mac and Windows computers.

GitHub
https://github.com/login
GitHub is of course where we store all our code for our website as we can easily make changes collaboratively and safely without erasing each other's work. We are all familiar with GitHub's interface and all have easy access to both the website version and the desktop version. Some of us are using the desktop version while the rest are using the website version purely just as a matter of preference.
